# 運用ガイド(人が読む・ファイルの使い方とフロー)

> この文書は**人が読む運用の入口**。`README.md` はプロジェクト概要(AI も読む可能性がある)に留め、
> 開発フローや各ファイルの使い方はこちらに置く。

## 📚 読む文書の使い分け
コンテキストの載り方は 3 層。**固定費(毎ターン常時ロード)を薄く保つ**のが軽さの要。

- **固定費(常時ロード)**: `CLAUDE.md`(+ import する `AGENTS.md`=規約)/ 各 skill の **description のみ** / MCP のツール定義(Learn・NuGet)。→ ここは薄く保つ。
- **オンデマンド(必要時だけ読まれる)**: `docs/**`(`architecture/*`・`README`・`reference`・`glossary`・`review-checklist` 等)と、commands・agents・skills の**本文**。読まれるのは次のいずれか:
  - command / skill / agent がその処理内で参照した時(例: `/review` が `review-checklist` と `architecture` を読む)
  - `AGENTS.md`/`CLAUDE.md` の指示で AI が実装・レビュー時に自発的に読む(「`docs/architecture` に従え」「`docs/README` を見よ」)
  - 人が**プロンプトで明示指定**した時
- **自動ロードされない(人向け)**: この `運用ガイド.md`・`docs/guides/*`、そして `README.md`。※ **`README.md` は Claude Code に自動ロードされない**ため、AI に常時効かせたいルールは README でなく `AGENTS.md`(と、そこが指す docs)へ置く。

## 🔄 開発ループ(要約)

```
決定→ /adr(Whyを残す)   実装(skill: csharp-layered-feature が自動ロード)
  → PostToolUse フックで dotnet format 検証(逆フィードバック)
  → /spec-sync(Web API は OpenAPI 再生成)  → 意図が変われば REQ 更新
  → /review(reviewer) + /cross-review(Codex)  → /done(DoDゲート)  → 人間が git commit
```

各段階で打つ**具体プロンプト**は `docs/guides/workflow.md`。立ち上げ手順は `docs/guides/getting-started.md`。

## 📁 ファイルは 4 種の動かし方で役割分担する
**人が呼ぶ**(`/`コマンド)/**自動で発火**(hook)/**AI が委譲**(subagent)/**必要時に自動ロード**(skill)。

### `.claude/` — 動かす仕組み

| ファイル | 動かす主体 | 役割 / いつ |
|---|---|---|
| `settings.json` | 自動(Claude Code が読む) | 権限と hooks の定義。`deny` で `docs/reference/**` の手編集を禁止 |
| `hooks/dotnet-verify.ps1` | 自動(PostToolUse:編集直後) | 変更が `.cs` 等なら `dotnet format` 検証→差分を Claude に返す(逆フィードバック) |
| `hooks/done-check.ps1` | 自動(Stop:応答終了時) | DoD を軽くリマインド |
| `commands/adr.md` | **人** `/adr <題>` | 決定(Why)を `docs/adr/` に追記 |
| `commands/spec-sync.md` | **人** `/spec-sync` | `docs/reference` を再生成(Web=OpenAPI/DB=スキーマ) |
| `commands/review.md` | **人** `/review` | `reviewer` サブエージェントを起動 |
| `commands/cross-review.md` | **人** `/cross-review` | Codex で別ベンダーのクロスレビュー手順を用意 |
| `commands/trace.md` | **人** `/trace` | ID 整合・リンク切れ・決定漏れを検査 |
| `commands/done.md` | **人** `/done` | 完了前の DoD ゲート |
| `commands/requirements.md` | **人** `/requirements <一言>` | 要求 `REQ-*` 草案(`requirements` に委譲) |
| `agents/requirements.md` | 委譲 | 要求(REQ)の草案 |
| `agents/reviewer.md` | `/review` から起動 | 原則適合+意図整合+トレーサビリティでレビュー |
| `agents/doc-sync.md` | `/spec-sync` 系 | コードから `docs/reference` を生成 |
| `skills/*/SKILL.md` | 自動ロード(関連時) | 実装・ADR・生成・git 提示の手順書 |

> **command と skill の違い**:command は「人が `/名前` で明示的に呼ぶ工程」。skill は「関連作業に入ると Claude が自動ロードする手順書」で、説明文だけ常時読まれ本文は必要時のみ。

### `docs/` — 何を読み・書き・生成するか

| 場所 | 寿命 | フロー内の使いどころ |
|---|---|---|
| `README.md` | — | 最初に読む。寿命クラス表＝どれを手で守り/生成/追記するかの契約 |
| `adr/` | 不変 | 決定のたびに `/adr` で追記。実装・レビューの「なぜ」の根拠 |
| `architecture/` | 長 | 実装(skill)とレビュー(reviewer)が従う原則。迷ったらここ |
| `requirements/` | 中 | 機能の入口。`REQ-*` に意図を書く |
| `glossary.md` | 長 | ドメイン用語(英語名併記)。命名の拠り所。コード/DB で分かる情報は書かない |
| `architecture/conventions.md` | 長 | プロジェクト固有方針(編集可)。`String`/`string` の使い分け・API 命名等 |
| `reference/` | 短(生成) | `/spec-sync` が生成。手編集禁止。現状仕様(OpenAPI 等)はここ |
| `review-checklist.md` | 中 | `/review`・`/cross-review` が共有するレビュー観点 |
| `guides/` | — | `getting-started.md`(立上げ)/`workflow.md`(回し方・具体プロンプト) |
| `traceability/index.md` | 派生 | `/trace` が更新。REQ↔ADR↔test↔code の対応 |

## 🔌 具体例:新しい API(機能)を 1 つ追加する
1. `/requirements <機能の一言>` → `docs/requirements/REQ-*.md` を草案 → 人が承認
2. 決定があれば `/adr <決定>` → `docs/adr/*.md` に追記
3. 実装:`csharp-layered-feature` スキルが自動ロード → `docs/architecture/` の責務に沿って
   Endpoint(minimal API)+ Service + 必要なら Component を実装
4. `/spec-sync` → `docs/reference/api/openapi.json` を再生成
5. 意図が変われば `docs/requirements/REQ-*.md` を同じ変更内で更新
6. `/review` (+ 別ベンダーで `/cross-review`) → `/done` → 人が `git commit`

## 📊 プロジェクト管理 (任意)
feature 単位のイテレーション計画・進捗が要るなら **`setup.ps1 -PM`** で有効化する(`/pm-plan`・`/pm-status`・`docs/pm/`)。無効なら関連ファイルは setup で削除される。

## 🧩 MCP と skill (拡張)

### MCP (標準同梱)
- `.mcp.json` に **Microsoft Learn**(公式ドキュメントで grounding)と **NuGet**(パッケージ・最新版・脆弱性修正)MCP を同梱。Claude Code の初回起動時に承認する。
- 種別が net10.0 なので NuGet MCP の前提 **.NET 10 SDK**(`dnx`)を満たす。CI でも .NET 10 を用意する。
- MCP は「ツール」で文書を生成・保持しない(二重管理の懸念なし)。
- **注意(コンテキスト)**: MCP の**ツール定義は固定費(常時ロード)**。**可視ツールが約 50 を超えると AI が誤ったツールを選びやすくなる**ため、MCP サーバは Learn / NuGet に絞る(用途が重なるものは 1 つに)。

### skill (標準同梱 + 任意で追加)
- 標準 skill(`csharp-layered-feature` / `write-adr` / `sync-docs-from-code` / `git-commit` / `distill-req`)は同梱済み。**テンプレに含めるのはここまで**。
- ASP.NET Core / .NET 向けの追加は 公式 **`dotnet/skills`**(Microsoft .NET チーム)や community キットから**選別**採用。導入は plugin marketplace か vendor-copy(LICENSE を添える)。
- **優先順位**: 外部 skill / MCP は参考・補完。**本プロジェクトの `docs/architecture/*` と `conventions.md` が常に優先**する(`AGENTS.md`・`conventions.md` に明記)。
- **注意(コンテキスト)**: skill の **description は固定費(常時ロード)**。本文は発動時のみだが、外部スキルを大量(例: 数十)に入れると description が全部載る + スキル / ツール過多で選択が乱れる。→ 必ず**選別して入れる**。
