# REQ 蒸留の手順(復元できる情報は残さない)

> 目的: REQ を「**コード / テスト / ADR / architecture から復元できない意図**」だけに絞る。実装が固まった後、意図の更新と同じ変更(PR)内で機械的に適用する。

## 判定: 各記述を 1 行ずつ 3 分類する
- **残す(REQ に置く)**: なぜこの機能が要るか(業務意図)/ 受け入れ条件を**意図として**/ スコープ・前提・制約(コードに現れないもの)/ 非機能の狙い。
- **移す**: 決定と理由 → `docs/adr/`(`/adr`)/ 一般原則 → `docs/architecture/`/ 用語 → `docs/glossary.md`。移した先は ID 参照にする。
- **落とす(REQ から消す)**: 実装詳細(引数・クラス名・手順)= コードで分かる / 具体的な現在の振る舞い = テスト・OpenAPI で分かる / 生成物と重複する記述。

## 手順
1. 対象 `REQ` を読み、各行に上の 3 ラベルを付ける。
2. 「移す」行を該当先へ反映し、REQ 側は ID 参照に置き換える(重複を残さない)。
3. 「落とす」行を削除する。
4. 残すのは非復元の意図のみ。frontmatter の `related`(ADR 等)を更新。
5. `/trace` で参照整合を確認。

## 判断基準(1 行)
> この行は **コード・テスト・ADR・architecture のどれかを見れば復元できるか?** できるなら REQ に残さない。
